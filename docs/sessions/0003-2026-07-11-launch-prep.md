### Phase 1 — Contrast fix, OG check, merge PR #1

**Achieved:**
- Merged PR #1 (`feat/site-2a-implementation`, all 10 phases from session 0002) into `main`.
- Fixed the "blue font hard to read in light theme" report. Root cause was **not** the token value itself — it was a CSS specificity bug in `brands.css`'s `@media(prefers-color-scheme:dark)` block, which had no `:not([data-theme="light"])` guard (unlike `foundation.css`'s equivalent `:root` block). A system-dark browser applied dark `.brand-skills` accent values even with `data-theme="light"` explicitly set — `#34D3EE` cyan on the light `--ds-bg` computed to ~1.6:1, not the ~3.3:1 the light token alone would give. Fixed with an ancestor-aware guard on the `@media` selectors (`html:not([data-theme="light"]) .brand-x`) plus swapping small/normal-text accent usages (kickers, micro-labels, `.ds-btn-quiet`, `.ds-link`, the 404 page, `About.astro`'s contact links) from `--ds-accent` to the darker `--ds-accent-hover` (already in the locked brand palette, no new hex). Left `--ds-accent` on backgrounds/borders/icons/large display text, where the 3:1 non-text/large-text bar already applied — reverted one over-application on Hero's italic emphasis (40-68px display type, already passed 3:1 before the fix).
- Verified OG/Twitter meta structurally (tags present, `og-image.png` exists at 1200×630) — live unfurl testing deferred to the manual checklist since it needs a resolvable URL, which didn't exist yet at this point in the session.

**Decisions:**
- Went through a 4-agent code-review pass (medium effort) before merging. Real findings: `src/pages/404.astro`'s scoped styles were missed entirely (only grepped `src/components/`, not `src/pages/`) — two independent finder agents caught this. `components.css`'s unused-but-latent `.ds-link`/`.ds-chip--accent` also fixed for consistency. One finder's proposed simplification (`.brand-skills:not([data-theme="light"])` directly on the class) was rejected — CSS-incorrect, since `:not()` checks the same element's own attributes and `data-theme` lives on `<html>`, not on the `.brand-skills` element; would have been a silent no-op.

### Phase 4 — Weekly updates page + scripts + highlight agent

**Achieved:**
- `scripts/stats/weekly_stats.py`: per-repo weekly LOC/sessions/top-tool/top-model (allowlist-filtered, same rule as Phase 3), a 7×24 hour-of-day/day-of-week activity heatmap, and PR-merge/release data per repo via `gh` CLI — titles only for repos `gh` reports as public, counts-only for private ones.
- `.claude/agents/weekly-highlights.md`: a narrow subagent (`Read`/`Edit`/`Glob` only, no `Bash`/`Write`) that paraphrases a week's already-computed JSON into short highlight prose, written back into the same file's `highlights` field. Structurally can't invent data or regenerate anything — its tool grant doesn't allow it.
- `src/pages/updates.astro`, `src/components/WeeklyHeatmap.astro`, `src/lib/weekly.ts` (+ types, unit tests): new page, separate from the all-time Telemetry section, listing one completed Mon-Sun week at a time, newest-first, reusing existing design tokens. Nav link added. Verified visually at 1440/1024/390 × light/dark.
- First real digest generated: `data/weekly/2026-W27.json` (Jun 29 – Jul 5).

**Decisions:**
- One code-review pass (8 candidate findings), all real, all fixed: a timezone inconsistency between the per-repo week-boundary check (compared raw UTC dates) and the heatmap (correctly localized first) — could silently drop or misattribute a session near midnight depending on UTC-offset direction; missing `try/except` around `gh pr list` JSON parsing (would crash the whole run on a malformed response, unlike the releases-parsing path three lines below which already tolerated this); a missing `white-space:nowrap` guard on the "busiest repo" stat tile (same class of bug already fixed elsewhere in the same file for a hyphenated model name); and a heatmap color/tooltip mismatch (color normalized on `out_tokens`, tooltip's quiet/active text branched on message count — a message with real activity but 0 recorded output tokens rendered as visually quiet while its tooltip said otherwise). Switched heatmap normalization to message count, which is also a better activity proxy than raw token volume for an hour-level chart.
- Two limitations accepted and documented rather than fixed: version bumps only show up via GitHub Releases, not bare git tags (tag dates need a second API call per tag — not worth it for a weekly job); a repo that goes private after its digest generated doesn't get retroactively redacted from already-committed past weeks.
