**Decisions:**
- **Cloudflare email obfuscation → plain `mailto:`.** The 2A source uses Cloudflare Pages' email-obfuscation feature (`__cf_email__` / `/cdn-cgi/l/email-protection#…`), which has no equivalent on GitHub Pages. Every occurrence (Hero, About, Footer) uses a plain, visible `mailto:contact@abhijitbansal.com` link instead. This is a necessary deviation forced by the hosting-platform change, not an oversight — the README's "Deviations for production" list doesn't cover it because it's Cloudflare-Pages-specific tooling that never had a GitHub-Pages equivalent to preserve.
- **`STATS_SOURCE` build-step design.** `scripts/fetch-stats.mjs` copies `data/stats.json` from `$STATS_SOURCE` when the env var is set and the file exists, else leaves the committed snapshot as-is, logging which path was taken. This repo does not own the telemetry generator (it lives in `claude-skills`); CI is expected to set `STATS_SOURCE` to point at freshly generated output. The committed snapshot keeps `npm run build` reproducible with no external dependency when `STATS_SOURCE` is unset.
- **Pixel-parity verification method.** The 2A `.dc.html` source cannot actually be opened via `file://` for a live side-by-side render — it links `_ds/house-design-system-.../styles.css` and a bundle script that aren't part of the design-handoff bundle (confirmed during Phase 10: opening it renders completely unstyled). Verification against the reference was and is done by transcribing inline style/hex/spacing values verbatim into the Astro components and design tokens, not by comparing two live renders. Worth flagging in the design-handoff bundle for future reference.

### Phase 10 — Full acceptance pass

**Achieved:**
- Re-ran the design-handoff README's acceptance checklist verbatim, in both themes, at 1440/1024/390, against the production build served via `astro preview`. Five real issues were found and fixed (see below); everything else passed clean on first or second pass.
- **Fix 1 — Telemetry copy-spacing bug** (`src/components/Telemetry.astro`): `Parsed locally from {filesFound}\n\t\t\tsession logs` collapsed to `"...2,884session logs"` — Astro (like JSX) drops whitespace-only text between an expression and a line-break-indented text node. Added an explicit `{' '}`. Verified fixed in both themes, matches the 2A source's `2,884 session logs` exactly.
- **Fix 2 — mobile nav overflow at 390px** (`src/components/Nav.astro`, `src/styles/global.css`): the 2A source has zero `@media` rules; its 7-item nav row (Work/Expertise/Telemetry/About/Theme/GitHub, all fixed-width mono text) overflows the viewport below ~520px with the overflow landing outside the fixed-position header's box — "Theme" and "GitHub" were genuinely unreachable (not just visually clipped; nothing scrolled to reveal them). Added `overflow-x: auto` + `min-width: 0` on `#fy-nav` under a `max-width: 600px` query, hidden scrollbar. Zero visual/copy change above that breakpoint; confirmed the full nav is reachable by horizontal swipe/scroll at 390px afterward.
- **Fix 3 — silent no-WebGL failure path** (`src/lib/three/init.ts`): when `createHeroScene` catches a `WebGLRenderer` construction failure internally and returns `null` (not a thrown exception), `initHeroScene` stored nothing and logged nothing — contradicting both the plan and the file's own header comment, which both call for a `console.warn` on any failure path. Added the missing `else` branch. Verified via a `getContext` override (forces WebGL creation to fail): poster stays visible, and `[foundry] scene init returned null (no WebGL?) — poster fallback stays` now logs.
- **Fix 4 — heading-order a11y gap** (`src/components/ProjectCard.astro`): Lighthouse flagged an invalid heading skip — the 2A source's project-card titles are `<h4>` directly under the Work section's `<h2>Forged here.</h2>`, with no `<h3>` in between. Changed to `<h3>` (both the standard-card and `layout="foundation"` variants); zero visual effect since typography is class-driven (`.ds-display-md`), not tag-driven.
- **Fix 5 — color-contrast a11y gap** (`src/components/Footer.astro`): the footer's "No trackers · No analytics · Nothing to consent to" line used `--ds-text-faint` (1.92:1 against `--ds-bg`, well under the 4.5:1 WCAG AA minimum for non-decorative text) — Lighthouse flagged it. Switched to `--ds-text-3` (4.66:1), the token already used one line above for the copyright text, so the two stay visually matched. `--ds-text-faint` is used the same non-decorative way elsewhere (Telemetry captions, "Private build" labels) but wasn't flagged by this Lighthouse pass — very likely because those elements are still `opacity:0` (pending scroll-reveal) at the single-snapshot audit's scan time. Not patched everywhere in this pass — see Decisions below.
- Re-ran `npm test` (40/40 green) and `npm run build` after every fix; re-ran the full checklist until clean.
- Verified: light-first default with no system-pref flash (fresh `localStorage`); dark re-palette live via the nav toggle, molten core staying orange in both themes; DPR cap (real DPR 3 → effective renderer DPR 2, confirmed via canvas-pixel/CSS-pixel ratio); render loop pauses when the hero scrolls out of view (rAF call count drops to 0, confirmed by instrumenting `requestAnimationFrame`); `prefers-reduced-motion` skips 3D init entirely (canvas never created, poster stays at its default opacity); no-WebGL fallback leaves the poster visible without throwing; all 9 resolved project links return `200` via `curl -I`; zero third-party requests beyond `fonts.googleapis.com`/`fonts.gstatic.com` on the production build (confirmed via a full network-request capture — the dev server's own tooling requests don't count, see Decisions); entry JS bundle ~4KB gz with `three.js` as its own ~170KB gz lazy chunk, both well inside/outside the 300KB pre-3D budget as intended; Lighthouse (mobile) — Accessibility 100, Best Practices 100, SEO 100 after the two a11y fixes; a performance trace showed LCP 96ms / CLS 0.06 on this machine (well inside the <2.5s / <0.1 targets, though this is a lab reading on an unthrottled local run, not a CI-conditions Lighthouse Performance category score — see Decisions); keyboard pass — skip link is the first Tab stop and jumps to `#work`, all 29 focusable elements are reachable in visual DOM order with no positive `tabindex` and no traps, `:focus-visible` outlines render on the skip link and other interactive elements; OG/Twitter meta, sitemap, favicons, and CNAME all present and correct in `dist/`.

**Decisions:**
- **Dev-server port collision caught mid-pass.** A stale `astro dev` process from an earlier, unrelated session already owned `localhost:4321`; my first `astro preview` invocations silently landed on `4322`/`4323` instead. Early screenshots/network captures were unknowingly against the **dev server**, not the production build — harmless for visual checks, but the "zero third-party requests" and bundle-size checks were re-run against the actual `astro preview` production server (port `4323`) to be valid. Worth remembering: always confirm which port/process actually answered before trusting a network-request capture.
- **`--ds-text-faint` contrast gap not patched everywhere.** Only the one Lighthouse-flagged Footer instance was fixed. The token is used the same non-decorative way in a few other places (Telemetry captions, "Private build" labels); whether to bump those too is a design-token-level call (this token is meant to come from the shared `design-system` repo per Task 1.1's vendoring note) rather than something to resolve unilaterally inside one component during an acceptance pass. Flagged as a Follow-up.
- **Hidden-tab render pause not conclusively verified by automation.** `src/lib/three/scene.ts`'s `syncRun()` deliberately does not gate on `document.hidden`/`pageVisible` — the code comment states it relies on the browser's native rAF throttling for background tabs, with the IntersectionObserver only handling the scroll-past-hero case. Attempting to verify this in the sandboxed/automated Chrome session used for this pass was inconclusive: `document.hidden`/`visibilityState` correctly read `true`/`'hidden'` after a real tab switch, but `requestAnimationFrame` kept firing at full rate regardless — automated/CDP-controlled tab backgrounding doesn't reliably reproduce the throttling a real user's OS-level tab switch triggers. This is a pre-existing, documented design decision (not touched this session) and the acceptance checklist item is better closed out with a real device/browser check — added to the manual-test checklist.
- **True side-by-side pixel-parity render against the 2A `.dc.html` source is not possible** with the assets in this repo (see Phases 0-9 Decisions above) — re-confirmed during Phase 10. Verification for this pass was against the transcribed token/spacing values plus visual-consistency review across breakpoints/themes, not a live-render diff.

**Follow-ups:**
- GitHub Pages settings + custom-domain DNS still need a human with repo-admin access (unchanged from Phases 0-9).
